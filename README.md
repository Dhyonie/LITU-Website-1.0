# LITU Website

Static site built from exports produced by the design tool's `x-dc` component
runtime.

## Structure

- `Home.dc.html`, `Services.dc.html`, `Contact.dc.html`, `Book.dc.html` — the
  four site pages (generated by the design tool, don't hand-edit — re-export
  and re-copy instead)
- `support.js` — the `dc-runtime` script that renders the `x-dc` templates
  (loads React/ReactDOM from unpkg on demand)
- `image-slot.js` — the `<x-import>` image-slot component used by Home and
  Services for their photo placeholders
- `assets/` — images (webp/png) referenced by the pages
- `index.html` — redirects `/` to `Home.dc.html` so the site root loads the
  homepage
- `exports/` — raw design-tool exports kept for reference (a `.zip` project
  archive and/or standalone bundled-page `.html` files), not part of the
  served site
- `scripts/unpack.mjs` — converts an older-style *single-file* "Bundled Page"
  export (a self-extracting HTML blob) into the same `<Name>.dc.html` +
  `assets/` structure. Only needed if the design tool gives you that format
  again instead of a project archive

## Updating pages

When you re-export from the design tool:

- **Project archive (.zip)** — the simplest case. Unzip it, then copy its
  `*.dc.html` files, `support.js`, `image-slot.js`, and `assets/*` into the
  project root (overwriting the existing ones). No build step needed.
- **Single bundled page (.html)** — drop the file into `exports/` and run
  `npm run unpack` to extract it into the same structure.

## Running locally

Everything is static HTML/CSS/JS — no build step for normal edits.

```sh
npm run dev
```

Serves the site at a local URL (via `serve`). Opening any `*.dc.html` file
directly from disk also works, but a local server avoids font-loading
quirks some browsers have with `file://` origins.

## Deploying

Pushing to `main` on GitHub auto-deploys via the connected Vercel project.
